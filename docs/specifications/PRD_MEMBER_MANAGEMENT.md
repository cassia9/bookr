# PRD：成員管理與權限系統

**版本:** 1.0  
**狀態:** Draft  
**最後更新:** 2026-06-03  
**優先級:** High

---

## 1️⃣ 問題陳述

### 現狀

目前預約系統只支持單一管理員帳號，無法多人協作。當店家有多名員工（按摩師、伸展師、美容師、前台等）時，無法：
- 邀請員工加入系統
- 區分員工角色和權限
- 管理員工的數據訪問權限
- 員工自助註冊並加入店家

### 為什麼重要

- **協作效率**：多名員工能同時使用系統，提高營運效率
- **數據安全**：不同角色只能訪問相關數據，防止誤操作和數據洩露
- **可擴展性**：為多店家架構奠定基礎
- **用戶體驗**：員工無需技術人員協助即可自助加入

---

## 2️⃣ 目標和成功指標

### 目標

建立一個成員管理系統，讓店家管理員能邀請員工、設定權限，員工能自助註冊並立即加入店家的協作環境。

### 成功指標

| 指標 | 基線 | 目標 | 衡量方法 |
|------|------|------|---------|
| 成員邀請完成度 | 0% | 管理員能在 <2 分鐘內邀請新成員 | 邀請流程計時測試 |
| 成員註冊自助率 | 0% | 100% 新成員能自助完成註冊和加入 | 無需協助的註冊次數 / 總次數 |
| 權限隔離有效性 | 不適用 | 一般成員無法訪問禁止的資料（SQL 測試） | RLS 政策單元測試通過率 100% |
| 管理功能覆蓋率 | 不適用 | 支持新增、編輯、刪除成員和權限調整 | 功能檢查清單 100% 完成 |

## 3️⃣ 解決方案概述

### 高層架構

```
管理員                      一般成員                        新員工
  ↓                          ↓                              ↓
Settings > 成員管理          個人資料                    邀請郵件中的連結
  ├─ 邀請成員 (郵件)          ├─ 查看角色                    ↓
  ├─ 管理成員列表              ├─ 修改密碼                  註冊頁面
  └─ 編輯/刪除成員            └─ 管理設定                    ↓
                                                          完成註冊
                                                          自動加入店家
```

### 核心功能

1. **成員邀請**
   - 管理員在設定頁面輸入成員郵件
   - 選擇角色（一般成員 / 管理員）
   - 系統發送邀請郵件（含唯一邀請連結）

2. **自助註冊**
   - 員工點擊邀請連結進入註冊頁面
   - 設定姓名、密碼
   - 完成後自動加入該店家，無需驗證

3. **成員管理**
   - 管理員查看所有成員列表
   - 編輯成員角色
   - 刪除成員（軟刪除，保留歷史記錄）

4. **權限控制**
   - 基於 Supabase RLS + 角色的訪問控制
   - 預訂、客戶、從業人員等表格的行級安全

## 4️⃣ 功能需求

### 4.1 管理員功能

#### FR-1: 邀請成員
- **描述**: 管理員可通過郵件邀請新成員加入店家
- **使用者故事**: 身為管理員，我想邀請新員工，這樣他們可以自助註冊並立即使用系統
- **驗收標準**:
  - ✅ 管理員可在 Settings > 成員管理 中輸入郵件和選擇角色
  - ✅ 發送邀請郵件，含唯一邀請連結（UUID）和店家名稱
  - ✅ 邀請連結有效期 30 天
  - ✅ 同一郵件不能重複邀請（檢查 pending_invitations 表）
  - ✅ 郵件發送失敗時顯示錯誤提示

#### FR-2: 查看成員列表
- **描述**: 管理員可查看所有已加入的成員及其角色
- **驗收標準**:
  - ✅ 顯示姓名、郵件、角色、加入日期、狀態
  - ✅ 支援排序和搜尋
  - ✅ 顯示活躍/非活躍狀態（基於 updated_at）

#### FR-3: 編輯成員角色
- **描述**: 管理員可更改成員的角色（一般成員 ↔ 管理員）
- **驗收標準**:
  - ✅ 提供下拉菜單選擇角色
  - ✅ 更改立即生效，無需重新登入
  - ✅ 記錄角色變更在審計日誌

#### FR-4: 刪除成員
- **描述**: 管理員可移除成員，該成員無法再訪問系統
- **驗收標準**:
  - ✅ 軟刪除（設置 deleted_at），保留歷史數據
  - ✅ 刪除前需確認提示
  - ✅ 刪除後成員無法登入
  - ✅ 該成員建立的預約、客戶資料保留（但該成員無法編輯）

#### FR-5: 取消待定邀請
- **描述**: 管理員可取消未被接受的邀請
- **驗收標準**:
  - ✅ 在邀請列表中顯示待定邀請
  - ✅ 可以重新發送或刪除邀請

### 4.2 一般成員功能

#### FR-6: 接受邀請並註冊
- **描述**: 員工通過郵件邀請連結完成自助註冊

## 6️⃣ 技術考慮

### 資料庫更新

#### 新表：pending_invitations
```sql
CREATE TABLE pending_invitations (
  id UUID PRIMARY KEY,
  store_id UUID NOT NULL,
  email TEXT NOT NULL,
  role user_role NOT NULL DEFAULT 'member',
  token UUID UNIQUE NOT NULL,
  created_by UUID NOT NULL,           -- 邀請人 ID
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ,             -- 30 days from creation
  accepted_at TIMESTAMPTZ,            -- 註冊完成時
  accepted_user_id UUID,              -- 新建的 user id
  UNIQUE(store_id, email)             -- 同一店同一郵件不重複
);
```

#### users 表修改
```sql
ALTER TABLE users
  ADD COLUMN invited_by UUID,         -- 邀請該成員的管理員 ID
  ADD COLUMN invited_at TIMESTAMPTZ;
```

### RLS 政策更新

#### pending_invitations 表
```sql
-- 管理員可查看和管理
CREATE POLICY "admin_manage_invitations"
  ON pending_invitations
  FOR ALL
  TO authenticated
  USING (store_id = current_store_id() AND is_admin());

-- 已接受的邀請對應成員可查看（選填）
CREATE POLICY "user_see_own_invitation"
  ON pending_invitations
  FOR SELECT
  TO authenticated
  USING (store_id = current_store_id() AND accepted_user_id = auth.uid());
```

#### users 表（現有 RLS 補充）
```sql
-- 一般成員只能修改自己的密碼和資料
CREATE POLICY "member_update_self"
  ON users
  FOR UPDATE
  TO authenticated
  USING (id = auth.uid())
  WITH CHECK (id = auth.uid() AND role = (SELECT role FROM users WHERE id = auth.uid()));

-- 管理員可修改其他成員的角色
CREATE POLICY "admin_update_member_role"
  ON users
  FOR UPDATE
  TO authenticated
  USING (store_id = current_store_id() AND is_admin() AND id != auth.uid())
  WITH CHECK (store_id = current_store_id() AND is_admin());
```

## 7️⃣ 依賴和風險

### 依賴

| 項目 | 說明 | 負責人 | 完成期 |
|------|------|--------|--------|
| 郵件服務配置 | SendGrid / Mailgun API 金鑰 | 開發團隊 | 需要立即 |
| Supabase RLS 更新 | 部署新的 RLS 政策 | 數據庫管理員 | 實施前 |
| 前端路由 | /auth/accept-invitation?token=UUID | 前端開發 | 同步進行 |
| 審計日誌系統 | 已在 SECURITY_IMPROVEMENT_PLAN.md 中規劃 | 開發團隊 | 優先完成 |

### 風險

| 風險 | 可能性 | 衝擊 | 緩解策略 |
|------|--------|------|---------|
| 郵件進入垃圾箱 | 中 | 中 | 使用可信任的郵件服務，設定 SPF/DKIM |
| Token 洩露被濫用 | 低 | 高 | Token 30天有效期，支持重新發送，Token 一次性 |
| 成員被誤刪 | 中 | 中 | 軟刪除 + 審計日誌 + 恢復機制（後續） |
| 角色提升攻擊 | 低 | 高 | RLS 政策嚴格檢驗，成員無法自改角色 |
| 邀請郵件延遲 | 低 | 低 | 提供「重新發送邀請」功能 |

### 假設

- 店家有有效的郵件地址用於成員聯繫
- 成員能訪問邀請郵件
- 30 天的邀請有效期足夠
